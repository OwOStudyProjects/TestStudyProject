## 📝 提交规范（重要）

为了提高代码可读性和团队协作效率，建议使用**规范化的 Commit Message**。

### 📌 基本格式

```bash
<type>: <简短描述>
```

示例：

```bash
feat: 添加用户登录功能
fix: 修复登录接口报错问题
chore: 更新依赖版本
```

---

### 📚 常用类型说明

| 类型         | 含义                               | 示例                      |
| ------------ | ---------------------------------- | ------------------------- |
| **feat**     | 新功能（feature）                  | `feat: 添加注册功能`      |
| **fix**      | 修复 bug                           | `fix: 修复支付失败问题`   |
| **docs**     | 文档修改                           | `docs: 更新 README`       |
| **style**    | 代码格式（不影响逻辑）             | `style: 调整代码缩进`     |
| **refactor** | 重构（既不是新功能也不是修复 bug） | `refactor: 优化登录逻辑`  |
| **test**     | 添加或修改测试                     | `test: 增加单元测试`      |
| **chore**    | 构建过程或辅助工具变动             | `chore: 更新依赖包`       |
| **perf**     | 性能优化                           | `perf: 提升接口响应速度`  |
| **build**    | 构建相关修改                       | `build: 修改打包配置`     |
| **ci**       | CI/CD 相关                         | `ci: 添加 GitHub Actions` |

---

### ✅ 编写建议

- 使用**中文或英文统一风格**
- 描述要**简洁明确**
- 一个 commit 只做一件事
- 避免使用模糊描述：

  - ❌ `修改代码`
  - ❌ `update`
  - ✅ `fix: 修复用户登录失败问题`
